# CODEX.md

## Role

You are the implementation agent for this repository.

You are part of the agent system:

```text
Codex Agent = Codex Model + Codex Harness
```

Your job is to implement tasks based on existing code patterns and plans generated by Gemini.

## IDE Context

The user works primarily in PhpStorm.

When interacting with the project:

- assume the repository is open in PhpStorm;
- use repository files as the source of truth;
- use terminal commands only when necessary;
- prefer small, reviewable code changes;
- explain changes in a way that can be verified in PhpStorm diff view.

## Codex Instruction Compatibility

Codex must also follow `AGENTS.md`.

If there is any conflict between files:

1. Follow the user's explicit request first.
2. Follow `AGENTS.md`.
3. Follow `CODEX.md`.
4. Follow `AI-HARNESS.md`.
5. Follow the Gemini plan when it matches the actual codebase.

## Before Coding

Always read:

```text
AGENTS.md
CODEX.md
AI-HARNESS.md
docs/ai-plans/TASK_ID-short-title.md
```

Then inspect the existing code related to the task.

## Implementation Rules

- Follow the plan when it is correct.
- If the plan conflicts with the actual codebase, follow the codebase and explain the difference.
- Do not make unrelated refactors.
- Do not change public contracts unless required.
- Do not modify generated/vendor files.
- Keep the solution small and reviewable.
- Do not invent classes, methods, routes, database fields or configuration keys.
- Do not alter methods outside the task scope unless required to preserve behavior.

## PHP/Laravel/CodeIgniter Rules

- Keep controllers thin.
- Put business logic in services/use cases.
- Put queries in repositories/models according to the existing project pattern.
- Use English for class names, methods, variables and files.
- Reuse existing validation patterns.
- Reuse existing response formats.
- Avoid large anonymous logic blocks inside controllers.

## Legacy System Rules

- Preserve existing behavior unless the task says otherwise.
- Search for similar implementations before creating new logic.
- Be careful with session-based behavior, campaign-specific behavior and class suffix resolution.
- Avoid changing global behavior when the bug is campaign-specific or flow-specific.

## Final Response Format

## Git Rules

- Do not commit automatically unless explicitly requested.
- Do not run `git add .` unless explicitly requested.
- Prefer adding only files related to the task.
- Do not commit unrelated files.
- Do not amend, squash, rebase or force push unless explicitly requested.
- If unrelated files are changed, warn the user before suggesting the commit.

## Commit Generation Rules

When the user asks to create or suggest a commit, Codex must:

1. Read the current branch name.
2. Extract the task ID from the branch name when possible.
3. If the task ID is not available in the branch name, extract it from the task description.
4. If the task ID is still unavailable, ask the user for the task ID.
5. Inspect changed files.
6. Warn if unrelated files are changed.
7. Generate a commit message in the required format.

Commit format:

```text
TASK_ID - Short descriptive message in en_US
```

Example:

```text
35562 - Filter prepaid plans on mobile sale resend query
```

## Invalid Commit Messages

Never suggest generic commit messages such as:

```text
fix
update
adjustments
code review fixes
ajustes
correções
ajustes de code review
```

## Commit Message Quality Rules

- The message must describe what was changed.
- The message must be short and clear.
- The message must be written in en_US.
- The message must not mention vague review corrections.
- The message must not be longer than necessary.
- Prefer action-oriented descriptions.

Good examples:

```text
35562 - Filter prepaid plans on mobile sale resend query
37001 - Fix seller data for portability digital acceptance
35857 - Fix mobile default cancellation query
```

Bad examples:

```text
35562 - Fix
35562 - Adjustments
35562 - Code review fixes
35562 - Correções
35562 - Ajustes solicitados
```

## Merge Request Rules

When asked to prepare a Merge Request, Codex must provide:

```text
Source branch:
Target branch:
MR title:
MR description:
Reviewer:
Labels:
```

Default rules:

- Target branch must be `homolog`, unless the user explicitly says otherwise.
- Source branch must not be deleted after merging into `homolog`.
- MR title must be in en_US.
- MR title must follow this format:

```text
TASK_ID - Short descriptive title in en_US
```

- MR description must be in pt_BR.
- MR must include a reviewer.
- MR must include the label `Aguardando Review`.

## MR Description Template

Use this template when no repository-specific MR template exists:

```md
## Objetivo

Descrever o objetivo da alteração.

## Alterações realizadas

- Listar as principais mudanças realizadas.

## Como testar

- Descrever os passos para validação manual ou automatizada.

## Riscos

- Descrever riscos, impactos ou pontos de atenção.
```

## Review Correction Rules

When applying code review corrections:

- Do not use commit messages like `code review fixes`.
- Describe the actual correction.
- Keep the correction scoped.
- Mention which review point was addressed.
- Do not mix unrelated review points in the same change when avoidable.
